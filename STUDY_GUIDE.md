# 代码质量研读指南：jest-validate

> 配套仓库：本目录的 `original/`（Jest 原始源码）+ `src/`（零依赖重构版）。
> 目标：通过研读一个**真实、知名、代码质量很高**的测试框架模块，提炼可落地的团队规范。

---

## 1. 模块定位

`jest-validate` 负责**校验 Jest 的配置对象**（如 `jest.config.js`）。当用户：

- 写错键名（拼写错误）
- 给某个键传了错误类型的值
- 使用了已废弃的配置项

它能给出**友好、可操作**的提示（含「你是不是想输入 X」）。它是 Jest「开发者体验（DX）」的关键细节之一——也是很多团队自己写配置校验时可以借鉴的范本。

---

## 2. 原始架构（`original/`）

| 文件 | 职责 |
| --- | --- |
| `validate.ts` | 对外入口 `validate(config, options)`，内部递归 `_validate` |
| `utils.ts` | `ValidationError` 错误类、颜色/格式化、`createDidYouMeanMessage`（拼写纠错） |
| `errors.ts` | `errorMessage` —— 拼装「类型不符」的友好文案 |
| `condition.ts` / `types.ts` / `defaultConfig.ts` | 类型与默认配置（本练习未逐字收录，不影响主线理解） |

**数据流向：**

```
validate(config, options)
   └─ 递归遍历 config 的每个键
        ├─ 是否废弃？      → deprecate() 回调
        ├─ 是否多类型特例？ → 内置特判（maxWorkers）
        ├─ 是否已知键？     → condition() 不符则 error() 回调
        ├─ 是否在黑名单？   → 跳过
        └─ 未知键          → unknown() 回调（含 Did-you-mean 提示）
```

设计亮点是**策略注入**：核心只管「遍历 + 分流」，具体怎么报错由调用方通过回调决定。这本身就是一条很好的解耦经验。

---

## 3. 我们发现的代码坏味道（Code Smells）

逐行研读后，挑出 4 个值得团队警惕的点：

### ① 反直觉的错误名：`this.name = ''`
`utils.ts` 的 `ValidationError` 构造函数里写了 `this.name = ''`。
错误类名被清空，排查问题时栈/日志里看不到有意义的名字。**好代码应保留 `name`。**

### ② 硬编码的魔法阈值：`< 3`
`createDidYouMeanMessage` 里 `leven(...) < 3` 的 `3` 直接写在表达式里。
想调参就得改逻辑、易遗漏。**好代码应抽成命名常量 `MAX_EDIT_DISTANCE`。**

### ③ 散落的魔法特例：`key === 'maxWorkers'`
`allowsMultipleTypes(key)` 内硬编码 `key === 'maxWorkers'`，且调用点埋在主流程里。
特例与通用逻辑混杂，难发现、难扩展。**好代码应把特例集中为显式白名单。**

### ④ 模块级可变状态：`let hasDeprecationWarnings = false`
`validate.ts` 用**模块级变量**在递归调用间共享状态。
在并发 / 重入场景下，上一次调用的结果会污染下一次（典型的「隐式全局状态」陷阱）。
**好代码应让函数纯净：状态通过返回值或局部对象传递。**

> 附带发现：原 `validate` 返回值里 `isValid` **恒为 `true`**——一个没有实际意义的「死字段」。
> 暴露恒真/恒假的字段会误导调用方，属于 API 设计中的「虚假承诺」，应删除或真正落地。

## 4. 我们的重构策略（`src/`）

| 策略 | 说明 |
| --- | --- |
| **零依赖** | 内联 `getType` / `leven` / `format` / 颜色，去掉 `chalk` `leven` `pretty-format` `@jest/get-type` 四个包，使模块无需 `npm install` 即可运行与单测 |
| **纯函数** | 去掉模块级可变变量，改用局部 `warnings` 对象收集状态 |
| **显式化** | 魔法特例 → `MULTI_TYPE_KEYS` 白名单；魔法阈值 → `MAX_EDIT_DISTANCE` 常量 |
| **可解释** | 中文注解标注「为什么」，把隐性知识显性化，便于团队 review |

重构后行为与原版一致（见 `examples/usage.js` 的三种场景）。

---

## 5. 可落地的团队规范建议

把以上提炼为团队评审清单（可写入 ESLint 规则 / PR 模板）：

1. **避免模块级可变状态**——并发与重入场景下极易出 Bug。
2. **不在 API 暴露恒真/恒假字段**——`isValid` 这种「死字段」应尽快删除或真正落地。
3. **魔法字符串 / 特例集中管理**——用命名常量或白名单，禁止散落主流程。
4. **错误信息要可操作**——不仅说「错了」，还要说「正确应该长什么样」「你是不是想输入 X」。
5. **依赖最小化**——能内联的几十行小工具，就不必引一个包（减少供应链与构建负担）。
6. **隐性知识显性化**——关键决策点写清「为什么」，降低后人维护成本。

## 6. 给团队的练习任务

1. 用 `node:test` 或 vitest 给 `src/` 写单元测试，覆盖：未知键 / 类型错误 / 废弃项三类。
2. 把 `isValid` **真正落地**（目前恒为 `true`）：任一错误即 `isValid = false`。
3. 给 `createDidYouMeanMessage` 加单测，验证阈值 `3` 的边界（距离 2 命中、距离 3 不命中）。
4. **互评实战**：每人挑一个 `src/` 文件做二次重构，开 PR 互相 review，把上面 6 条规范用起来。

> 通过「读名项目 → 找坏味道 → 动手重构 → 互评」的闭环，团队的代码审美与质量把控能力会快速提升。
